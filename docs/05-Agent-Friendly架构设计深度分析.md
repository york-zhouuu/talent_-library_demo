# Agent-Friendly 人才库架构设计深度分析

## 核心问题

如何设计一个既能服务人类用户，又能被 Agent 高效调用的人才库系统？

---

## 一、设计哲学：API-First + 双轨制

### 1.1 传统系统 vs Agent-Friendly 系统

```
传统系统：
  UI → 后端 API → 数据库
  (API 是为 UI 服务的，往往耦合 UI 逻辑)

Agent-Friendly 系统：
  ┌─────────────────────────────────────┐
  │           统一的 API 层              │
  │  (API 是独立的，不依赖任何调用方)      │
  └─────────────────────────────────────┘
           ↑                    ↑
     Human UI               Agent
  (飞书/微信/APP)      (OpenClaw/MCP/其他)
```

### 1.2 核心原则

| 原则 | 说明 |
|------|------|
| **API First** | 先设计 API，再构建 UI；API 不为特定调用方设计 |
| **语义优先** | 支持自然语言输入，结构化输出 |
| **可组合性** | 原子化 API，让 Agent 自由组合 |
| **自描述性** | 响应包含足够上下文，Agent 能理解结果 |
| **可观测性** | 完整的调用链路追踪和反馈机制 |

### 3.3 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 向量数据库 | pgvector (而非独立向量库) | 简化架构，事务一致性 |
| 缓存 | Redis | 成熟稳定，支持多种数据结构 |
| AI 模型 | OpenAI API | 效果好，迭代快，可替换 |

```python
# 多轮对话上下文管理

class SearchContext:
    """搜索会话上下文"""

    session_id: str

    # 累积的搜索条件
    accumulated_filters: dict = {
        "cities": ["北京"],           # 第1轮: "北京的"
        "min_experience": 5,          # 第2轮: "5年以上"
        "skills": ["Go"],             # 第3轮: "会Go"
        "max_salary": 50,             # 第4轮: "50万以内"
    }

    # 历史查询
    query_history: list = [
        "北京的工程师",
        "5年以上",
        "会Go",
        "50万以内"
    ]

    # 当前结果集
    current_results: list[str]  # candidate_ids

    # 用户反馈
    feedbacks: list = [
        {"candidate_id": "xxx", "type": "like", "reason": "大厂背景"},
        {"candidate_id": "yyy", "type": "dislike", "reason": "经验不足"},
    ]
```

## 七、关键设计决策

### 7.1 为什么要分 Human API 和 Agent API？

| 维度 | Human API | Agent API |
|------|-----------|-----------|
| 输入 | 表单/UI 交互 | 结构化 JSON |
| 输出 | 渲染友好 | 解析友好 |
| 错误处理 | 用户友好提示 | 机器可解析错误码 |
| 批量操作 | 少 | 多 |
| 上下文 | UI 状态管理 | 显式传递 |

### 7.2 为什么需要 MCP Server？

- **标准化**: MCP 是 Anthropic 推出的标准协议，未来会有更多 Agent 框架支持
- **可发现性**: Agent 可以自动发现可用工具
- **跨框架**: 不仅限于 OpenClaw，其他 Agent 也能接入

### 7.3 为什么 Memory 分三层？

```
短期 (Session)  →  解决多轮对话问题
中期 (User)     →  解决个性化问题
长期 (Global)   →  解决推荐优化问题
```

## 九、总结

Agent-Friendly 人才库的核心是：

1. **基建层**: 稳定的数据存储 + 强大的 AI 服务
2. **中间件层**: 语义理解 + 上下文管理 + Memory
3. **应用层**: 双轨制 (Human UI + Agent API)

关键成功因素：
- API 设计的原子化和可组合性
- 完善的上下文和 Memory 管理
- 反馈闭环驱动的持续优化
