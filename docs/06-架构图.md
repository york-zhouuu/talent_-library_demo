## 一、系统分层架构

### 1.1 整体架构分层

系统采用分层架构，但**不强制所有请求走完整路径**。根据操作复杂度，请求可以选择最短有效路径。

```mermaid
flowchart TB
    subgraph L1["Layer 1: 接入层 (Access Layer)"]
        direction LR
        H1[飞书插件]
        H2[Web Dashboard]
        H3[独立 APP]
        A1[MCP Server]
        A2[Skills API]
        A3[REST API]
        Router[路由决策]
    end

    subgraph L2["Layer 2: 上下文层 (Context Layer) - 可选"]
        direction LR
        JC[Job Context<br/>职位上下文]
        CKB[Candidate Knowledge Base<br/>候选人知识库]
    end

    subgraph L3["Layer 3: 业务服务层 (Service Layer) - 可选"]
        direction LR
        SS[Search Service<br/>搜索服务]
        ES[Evaluation Service<br/>评估服务]
        CS[Communication Service<br/>沟通服务]
        IS[Import Service<br/>导入服务]
    end

    subgraph L4["Layer 4: 基础设施层 (Infrastructure Layer)"]
        direction LR
        AI[AI Service<br/>LLM/Embedding]
        DB[(PostgreSQL)]
        CACHE[(Redis)]
        STORAGE[(Object Storage)]
    end

    %% 路由决策
    Router -->|"Full Path<br/>复杂查询"| L2
    Router -->|"Service Path<br/>业务操作"| L3
    Router -->|"Direct Path<br/>简单查询"| L4

    %% 层间依赖
    L2 --> L3
    L2 --> L4
    L3 --> L4

    %% 样式
    classDef humanAccess fill:#e1f5fe,stroke:#01579b
    classDef agentAccess fill:#f3e5f5,stroke:#7b1fa2
    classDef context fill:#fff3e0,stroke:#ef6c00
    classDef service fill:#e8f5e9,stroke:#2e7d32

    classDef router fill:#ffeb3b,stroke:#f57f17

    class H1,H2,H3 humanAccess
    class A1,A2,A3 agentAccess
    class JC,CKB context
    class SS,ES,CS,IS service
    class AI,DB,CACHE,STORAGE infra
    class Router router
```

### 1.2 层级职责定义

| 层级 | 名称 | 职责 | 何时需要 | 可跳过场景 |
|------|------|------|----------|-----------|
| L1 | 接入层 | 协议转换、认证、路由 | 始终需要 | 不可跳过 |
| L2 | 上下文层 | 语义理解、意图解析、上下文管理 | 自然语言查询、多轮对话 | 简单关键词搜索、精确查询 |
| L3 | 业务服务层 | 业务逻辑、权限校验、结果组装 | 复杂业务操作 | 简单数据读取 |
| L4 | 基础设施层 | 数据存储、AI 能力 | 始终需要 | 不可跳过 |

### 2.3 上下文层数据生命周期

| 数据类型 | 创建时机 | 更新时机 | 销毁时机 | 可重算 |
|----------|----------|----------|----------|--------|
| Job Context | 用户输入 JD/描述 | 用户修正 | 搜索结束 | ✅ |
| CKB.Source | 简历导入 | 用户手动修改 | 候选人删除 | ❌ |
| CKB.Profile | 导入后自动生成 | AI 模型升级 | Source 删除 | ✅ |
| CKB.Knowledge | 首次交互 | 每次交互追加 | 永不销毁 | ❌ |
| CKB.Session | 搜索开始 | 搜索过程中 | 会话超时 | 不适用 |

### 9.2 依赖规则

1. **禁止循环依赖**: A→B→C→A 形式的依赖链不允许存在
2. **按需走层，非强制全路径**: 根据操作复杂度选择最短有效路径
3. **上下文层可选**: 简单查询可跳过，复杂查询需要
4. **基础设施无业务逻辑**: L4 不包含业务逻辑，只提供能力

## 十、路由策略：按需走层

### 10.1 核心原则

**不是所有请求都必须走 L1→L2→L3→L4，而是根据操作复杂度选择最短有效路径。**

层的价值：
- **L2 上下文层**: 语义理解、意图解析、上下文管理
- **L3 服务层**: 业务逻辑、权限校验、结果组装
- **L4 基础设施**: 数据存储、AI 能力

如果某个操作不需要某层提供的价值，就不应该强制经过该层。

### 10.2 路由决策图

```mermaid
flowchart TB
    Start[请求进入] --> Q1{需要语义理解?}

    Q1 -->|是| Q2{需要上下文累积?}
    Q1 -->|否| Q3{需要业务逻辑?}

    Q2 -->|是| Full["Full Path<br/>L1→L2→L3→L4"]
    Q2 -->|否| Semantic["Semantic Path<br/>L1→L2→L4"]

    Q3 -->|是| Service["Service Path<br/>L1→L3→L4"]
    Q3 -->|否| Direct["Direct Path<br/>L1→L4"]

    subgraph Examples["示例"]
        E1["'有大模型经验的PM' → Full Path"]
        E2["'Python 北京' → Direct Path"]
        E3["GET /candidates/123 → Direct Path"]
        E4["继续筛选上一轮结果 → Full Path"]
    end
```

### 10.4 路径选择矩阵

| 操作类型 | 路径 | 经过的层 | 原因 |
|----------|------|----------|------|
| 自然语言搜索 | Full | L1→L2→L3→L4 | 需要意图解析 + 关键词扩展 + 排序 |
| 多轮对话搜索 | Full | L1→L2→L3→L4 | 需要上下文累积 |
| 候选人评估 | Full | L1→L2→L3→L4 | 需要职位上下文 + AI 分析 |
| 简单关键词搜索 | Direct | L1→L4 | 关键词明确，直接匹配 |
| 精确字段查询 | Direct | L1→L4 | 已是结构化条件 |
| 获取候选人详情 | Direct | L1→L4 | 简单 ID 查询 |
| 批量导入简历 | Service | L1→L3→L4 | 需要业务逻辑，不需要搜索上下文 |
| 记录沟通反馈 | Service | L1→L3→L4 | 需要业务校验 |
| 语义相似搜索 | Semantic | L1→L2→L4 | 需要语义理解，不需要复杂业务逻辑 |

### 10.5 API 设计示例

```yaml
# Direct Path: 简单关键词搜索
POST /api/v1/search/keywords
  path: direct
  layers: [L1, L4]
  input:
    keywords: ["Python", "北京"]
    filters:
      min_experience: 3
  behavior: 直接构造 SQL 查询，不经过语义解析

# Full Path: 自然语言搜索
POST /api/v1/search/natural
  path: full
  layers: [L1, L2, L3, L4]
  input:
    query: "有大模型经验的产品经理，最好在北京"
    session_id: "xxx"  # 可选，用于多轮对话
  behavior:
    - L2 解析意图，扩展关键词
    - L3 执行多轮搜索，排序
    - L4 查询数据库

# Service Path: 批量导入
POST /api/v1/import/batch
  path: service
  layers: [L1, L3, L4]
  input:
    files: [...]
  behavior:
    - L3 处理文件解析、去重、校验
    - L4 存储数据、生成画像

# Direct Path: 获取详情
GET /api/v1/candidates/{id}
  path: direct
  layers: [L1, L4]
  behavior: 直接查询数据库返回
```

### 10.6 路由决策的实现

```mermaid
flowchart TB
    subgraph Router["路由层 (在 L1 中)"]
        R1[解析请求类型]
        R2[判断复杂度]
        R3[选择路径]
        R4[执行调用]

        R1 --> R2 --> R3 --> R4
    end

    subgraph Rules["路由规则"]
        Rule1["endpoint 包含 /natural → Full Path"]
        Rule2["endpoint 包含 /keywords → Direct Path"]
        Rule3["method = GET + 简单 ID → Direct Path"]
        Rule4["需要 session_id → Full Path"]
    end

    R2 --> Rules
```

### 10.7 跨层调用的权限控制

虽然允许跨层调用，但仍需权限控制：

| 路径 | 默认权限 | 说明 |
|------|----------|------|
| Full Path | 所有用户 | 标准路径，完整功能 |
| Service Path | 所有用户 | 跳过上下文，但有业务校验 |
| Semantic Path | 所有用户 | 跳过服务层，但有语义理解 |
| Direct Path | 需声明 | 跳过上下文和服务，需要说明原因 |
| Raw SQL | 仅 Admin | 直接操作数据库，审计记录 |

### 10.8 为什么这样设计

| 设计决策 | 理由 |
|----------|------|
| **按需走层** | 避免简单操作的性能开销 |
| **路径可选** | 给 Agent 灵活性，不强制走固定流程 |
| **保留完整路径** | 复杂操作仍然需要完整的语义理解和业务逻辑 |
| **权限控制** | 灵活不等于无序，仍需要审计和控制 |
